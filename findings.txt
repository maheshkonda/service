OUR RECOMMENDATION
==================
Geo-DR Alias Pairing - Native Azure Premium feature

This directly addresses both contexts you raised in the meeting:

1. Configuration sync:
   Geo-DR alias pairing handles this natively without any pipeline 
   involvement. Once paired, all entity metadata - queues, topics, 
   subscriptions, authorization rules, filters - replicates 
   automatically and continuously from primary to BCP. Any new queue 
   or topic added to primary in the future will automatically appear 
   in BCP. No manual intervention needed.

2. Data sync (messages):
   The TTO platform is a document processing workload. Every tax 
   document uploaded by users is stored in Azure Blob Storage which 
   already has geo-redundant replication enabled - meaning the actual 
   source files exist in both East US 2 and Central US at all times.

   The Service Bus messages are not the documents themselves - they 
   are processing instructions like "go process document X" or 
   "document X scan is complete, proceed to extraction". If East US 2 
   goes down and some of these in-flight instructions are lost, the 
   source documents are still safely in BCP blob storage and can be 
   reprocessed from scratch.

   We therefore do not recommend building a message replication layer 
   on top of the alias pairing for this workload. Message replication 
   would require every microservice across the platform to be rebuilt 
   to handle receiving duplicate messages, introduces message ordering 
   risks in the processing pipeline, adds 24/7 operational overhead, 
   and all of this for an event that may never happen in the lifetime 
   of the platform. The cost does not justify the benefit here.

Pairing plan:
azuse2shdprdsbns01 --> azuscshdbcpsbns01  alias: shdprdsbns01-dr-alias
azuse2shdprdsbns02 --> azuscshdbcpsbns02  alias: shdprdsbns02-dr-alias

What failover looks like:
When East US 2 goes down, one CLI command promotes the BCP namespace 
to primary under the same alias. Applications reconnect automatically 
without any connection string changes because they were always talking 
to the alias, not the direct namespace endpoint. This fits cleanly 
into Stage 7 of the existing BCP failover pipeline with the manual 
approval gate already in place.
